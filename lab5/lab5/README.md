# Лабораторная работа №5 — Кеширование данных и управление сессиями с использованием Redis

Проект продолжает Лабораторную работу №4 (Go + Gin + GORM + PostgreSQL, JWT в HttpOnly cookies,
OAuth через Yandex/VK, CRUD ресурсов с проверкой владения, Soft Delete, Swagger UI). В этой лабе
к существующему стеку добавлен **Redis 7** для:

- кеширования часто читаемых данных (списки items, конкретный item, профиль `/auth/whoami`)
- **реестра JTI выпущенных access-токенов** — для мгновенного отзыва доступа через `/auth/logout`
  до естественного истечения JWT.

Весь существующий функционал (регистрация, JWT, OAuth, CRUD, миграции, Swagger) **остался
работоспособен** — кеш встроен как опциональный слой: если Redis недоступен, приложение продолжает
работать, просто без ускорения.

## Архитектура кеширования

### Стратегия — Cache-Aside (Lazy Loading)

Логика по любому `GET`-запросу:

1. Сформировать ключ кеша (с префиксом, см. ниже).
2. Прочитать из Redis. Если есть (**Cache Hit**) — вернуть.
3. Если нет (**Cache Miss**) — сходить в БД.
4. Положить результат в Redis с TTL.
5. Вернуть клиенту.

На любом `POST`/`PUT`/`PATCH`/`DELETE` производится **явная инвалидация** связанных ключей.
Логика построения ключей и инвалидации лежит в коде сервисов (`services/items_service.go`,
`services/auth_service.go`) — это требование задания: формирование ключей не должно быть скрыто
в библиотеке/конфиге.

### Модуль `cache/`

Отдельный пакет, инкапсулирующий работу с Redis.

```go
type Service struct { /* ... */ }

func (s *Service) Get(ctx, key, dst) error
func (s *Service) Set(ctx, key, value, ttl) error
func (s *Service) GetString(ctx, key) (string, error)
func (s *Service) SetString(ctx, key, value, ttl) error
func (s *Service) Exists(ctx, key) (bool, error)
func (s *Service) Del(ctx, keys...) error
func (s *Service) DelByPattern(ctx, pattern) error
```

Особенности реализации:

- **Сериализация — JSON** (для удобной читаемости значений через `redis-cli GET`).
- **TTL обязателен**: если в `Set` передан `0` — берётся `CACHE_TTL_DEFAULT` из конфига.
- **`DelByPattern`** реализован через `SCAN + UNLINK`, а не через `KEYS + DEL` —
  потому что `KEYS` блокирует Redis (один из контрольных вопросов лабы как раз про это).
  Дополнительно паттерн обязан начинаться с `wp:`, чтобы исключить случайное массовое удаление.
- **Graceful degradation**: если Redis недоступен (нет коннекта при старте или таймаут на лету),
  методы возвращают `ErrCacheUnavailable`, бизнес-логика трактует это как cache miss
  и работает напрямую с БД. Приложение **не падает**.

### Схема ключей

Все ключи приложения начинаются с префикса `wp:` (`AppPrefix` в коде):

| Назначение | Ключ | Где формируется |
|-----------|------|-----------------|
| Страница списка items пользователя | `wp:items:user:{userId}:list:page:{p}:limit:{l}` | `cache.KeyItemsList` |
| Конкретный item | `wp:items:user:{userId}:item:{itemId}` | `cache.KeyItemByID` |
| Профиль пользователя (для `/whoami`) | `wp:users:profile:{userId}` | `cache.KeyUserProfile` |
| Активный JTI access-токена | `wp:auth:user:{userId}:access:{jti}` | `cache.KeyAccessJTI` |

Паттерны для массовой инвалидации:

| Назначение | Паттерн |
|-----------|---------|
| Все страницы списка items пользователя | `wp:items:user:{userId}:list:*` |
| Все access JTI пользователя (для `logout-all`) | `wp:auth:user:{userId}:access:*` |

### Поведение эндпоинтов

| Метод | URI | Действие с кешем |
|------|-----|------------------|
| `GET` | `/items` | Cache-Aside: ключ списка с TTL |
| `GET` | `/items/:id` | Cache-Aside: ключ конкретного item с TTL |
| `POST` | `/items` | Инвалидация всех страниц списка владельца |
| `PUT` | `/items/:id` | Инвалидация списка + конкретного item |
| `PATCH` | `/items/:id` | То же |
| `DELETE` | `/items/:id` | То же |
| `GET` | `/auth/whoami` | Cache-Aside: ключ профиля с TTL |
| `POST` | `/auth/login`, `/register`, OAuth callback | Запись JTI access-токена в Redis |
| `POST` | `/auth/logout` | Удаление JTI текущего access + кеша профиля |
| `POST` | `/auth/logout-all` | Удаление всех JTI access юзера (по паттерну) + кеша профиля |

## Управление access-токенами через Redis (JTI registry)

Чистый JWT — stateless: после выдачи отозвать его невозможно, токен живёт до `exp`.
Чтобы реализовать **мгновенный** `/auth/logout` (закрытие текущего сеанса), используется
комбинация подходов:

1. В момент выпуска access-токена (`AuthService.IssueTokens`) его уникальный идентификатор
   (claim `jti`, UUID) сохраняется в Redis:

   ```
   SET wp:auth:user:{userId}:access:{jti}  "valid"  EX <accessExpirationSeconds>
   ```

   TTL ровно равен времени жизни JWT — Redis сам подчистит ключ, когда токен и так истечёт.

2. На каждом защищённом запросе `middleware/auth.go`:
   - проверяет подпись JWT (как и раньше);
   - **дополнительно** делает `EXISTS` ключа в Redis. Если ключа нет — 401, даже если подпись
     валидна и срок не вышел.

3. `POST /auth/logout` делает `DEL` на этот конкретный JTI (и заодно инвалидирует кеш профиля
   и отзывает refresh в БД). С этого момента старый access мгновенно перестаёт работать.

4. `POST /auth/logout-all` делает `DelByPattern` на `wp:auth:user:{id}:access:*` — выкидывает
   юзера со всех устройств.

**Что хранится в Redis для токенов:**

- ✅ только идентификатор (JTI) — короткий UUID без секретных данных;
- ❌ **не хранится** сам access-токен, его подпись, claims, и тем более пароль.

Это прямое требование задания: «Запрещено хранение полных токенов доступа в открытом виде
(хранить JTI или хеш)».

**Fail-open для отказоустойчивости.** Если Redis отвалился на лету, метод
`IsAccessJTIValid` возвращает `true` — т.е. middleware пропускает запрос (подпись JWT
всё ещё проверяется). Альтернатива — fail-closed — превратила бы Redis в SPOF. Это
осознанный trade-off; на проде он может быть пересмотрен.

## Безопасность

| Требование задания | Где реализовано |
|--------------------|-----------------|
| Подключение к Redis по паролю | `docker-compose.yml`: `--requirepass ${REDIS_PASSWORD}`; клиент в `cache.New` передаёт пароль из `.env` |

| Запрет хранения полных токенов | в Redis уходит только JTI, не сам access |

| `SCAN` вместо `KEYS` | в `Service.DelByPattern` используется `SCAN` + `UNLINK` |

## Сценарий ручного тестирования

```bash
# 0. Старт
docker compose up --build

# 1. Регистрация (создаёт юзера + сразу логинит, выставляет cookies)
curl -i -c cookies.txt -X POST http://localhost:4200/auth/register \
  -H 'Content-Type: application/json' \
  -d '{"email":"alice@example.com","password":"StrongPass123"}'

# 2. Создаём пару items
curl -b cookies.txt -X POST http://localhost:4200/items \
  -H 'Content-Type: application/json' \
  -d '{"name":"Молоток","description":"стальной"}'

curl -b cookies.txt -X POST http://localhost:4200/items \
  -H 'Content-Type: application/json' \
  -d '{"name":"Отвёртка","description":"крестовая"}'

# 3. Первый GET /items — Cache Miss, в логах "[items] MISS ... — запрос в БД"
curl -b cookies.txt http://localhost:4200/items?page=1\&limit=10

# 4. Повторный GET — Cache Hit, "[items] HIT ...". В Redis должен лежать ключ:
docker exec -it wp_labs_redis redis-cli -a "$REDIS_PASSWORD" \
  KEYS 'wp:items:*'

# 5. Создаём ещё один item — ключ списка должен исчезнуть
curl -b cookies.txt -X POST http://localhost:4200/items \
  -H 'Content-Type: application/json' \
  -d '{"name":"Пила","description":"ножовка по дереву"}'

docker exec -it wp_labs_redis redis-cli -a "$REDIS_PASSWORD" \
  KEYS 'wp:items:*'   # ключа списка больше нет

# 6. /whoami — кешируется профиль
curl -b cookies.txt http://localhost:4200/auth/whoami
docker exec -it wp_labs_redis redis-cli -a "$REDIS_PASSWORD" \
  KEYS 'wp:users:profile:*'

# 7. JTI access токена тоже в Redis:
docker exec -it wp_labs_redis redis-cli -a "$REDIS_PASSWORD" \
  KEYS 'wp:auth:user:*:access:*'

# 8. Logout — JTI должен пропасть, и повторный запрос со старым cookie должен дать 401
curl -b cookies.txt -X POST http://localhost:4200/auth/logout
curl -b cookies.txt http://localhost:4200/auth/whoami      # -> 401, если cookie не была сброшена
```

В шаге 8 примечание: `curl -b cookies.txt` сам по себе не подхватит обновлённые `Set-Cookie`
от ответа `/logout` без `-c cookies.txt`. Корректнее протестировать «отзыв до истечения JWT»
так: после `/auth/login` записать access-токен и затем сделать `/auth/logout` отдельным
запросом с тем же токеном — следующий запрос с этим access вернёт 401, потому что JTI
удалён из Redis.

## Контрольные вопросы

**1. В чём разница между стратегиями Cache-Aside и Write-Through?**

В **Cache-Aside (Lazy Loading)** приложение работает с кешем явно: при чтении сначала смотрит
в кеш, при промахе идёт в БД и сам кладёт результат; при записи — обновляет БД и инвалидирует
ключ. Кеш не знает про БД и наоборот. Плюс — простота и устойчивость: падение кеша не ломает
запись. Минус — первая чтение после промаха медленнее, и до записи в кеш данные туда не
попадут (cold cache).

В **Write-Through** запись идёт через кеш: приложение пишет в кеш, кеш сам обновляет БД
синхронно. Чтения почти всегда hit, согласованность по записи сильнее. Минус — записи медленнее
(две операции вместо одной), сложнее в реализации, кеш становится критическим компонентом —
его падение блокирует и БД-запись тоже.

В этом проекте использован Cache-Aside — он лучше подходит к требованию «приложение не должно
падать при недоступности Redis».

**2. Что такое "Thundering Herd problem" и как кеширование может помочь или усугубить её?**

Thundering Herd — это когда в один момент истекает популярный ключ кеша, и десятки/сотни
параллельных запросов одновременно делают cache miss и идут к БД за одним и тем же.
БД получает шквал одинаковых запросов и может лечь.

Кеширование **порождает** эту проблему (без кеша запросы были бы распределены постоянной
нагрузкой). Смягчают её:

- **Stale-while-revalidate**: на промахе ровно один воркер идёт в БД, остальные пока отдают
  устаревшее значение.
- **Single-flight / mutex per key**: первый запрос лочит ключ, остальные ждут результата.
- **Jittered TTL**: рандомизировать TTL на ±10–20%, чтобы ключи протухали в разное время.
- **Refresh-ahead**: фоновое обновление за N секунд до истечения.

В текущей лабе ни один из этих приёмов не реализован — задание этого не требует, но в
проде их стоит добавить, особенно single-flight.

**3. Почему не рекомендуется использовать `KEYS *` в продакшене?**

`KEYS` — **блокирующая** операция, она проходит по всему keyspace в один проход и блокирует
single-thread Redis на всё это время. На сотнях тысяч / миллионах ключей это сотни миллисекунд
паузы для всех клиентов (включая healthcheck-и, что может вызвать каскадные таймауты).
Дополнительно `KEYS *` строит огромный ответ в памяти.

Правильная замена — `SCAN` с курсором: он отдаёт небольшими батчами, между батчами Redis
обслуживает другие клиенты. В этом проекте `cache.Service.DelByPattern` именно так и делает:
`SCAN ... COUNT 100` + `UNLINK` (асинхронное удаление, не блокирует event loop).

**4. Как обеспечить согласованность данных между БД и кешем при одновременной записи?**

Полностью линейной согласованности не добиться без двухфазного коммита между БД и кешем.
Практические подходы:

- **Cache-Aside с инвалидацией после записи в БД** (как в этом проекте). Окно несогласованности
  существует, но короткое — от commit'а БД до `DEL` ключа.
- **Delete, not update**: при записи именно удалять ключ, а не обновлять. Тогда параллельный
  читатель в худшем случае подтянет свежее значение, а не перезапишет более старым своим
  результатом поверх свежего (классическая гонка writer-reader-writer).
- **Write-Through / Write-Behind**: пишем через кеш, он гарантирует порядок.
- **Версионирование значений** в кеше (например, в значение писать `updated_at` из БД).
- **CDC / outbox + Debezium**: вынести инвалидацию в фоновый процессор, читающий WAL Postgres.
- На уровне сервиса в этом проекте дополнительно: операции записи (`Update`, `Patch`, `Delete`)
  читают item напрямую из БД, минуя кеш (`getByIDFromDB`), чтобы не сохранить устаревшие поля
  поверх свежих.

**5. Зачем нужен TTL у ключей кеша, если мы реализуем инвалидацию при записи?**

TTL — это страховка. Инвалидация может не сработать по куче причин: упал сервис между
коммитом в БД и `DEL`; неучтённый источник записи (миграция, фоновый job, ручной SQL в
прод-БД); баг в коде; партиция сети между приложением и Redis. Без TTL такие «зависшие»
ключи живут вечно и отдают устаревшие данные. С TTL — максимум `TTL` минут несогласованности.
Плюс TTL естественно ограничивает рост памяти Redis.

**6. Почему хранение JTI Access токена в Redis позволяет реализовать мгновенный Logout,
несмотря на Stateless природу JWT?**

Stateless у JWT означает «сервер не обязан хранить состояние, чтобы проверить токен» —
достаточно подписи и `exp`. Это и плюс (горизонтальное масштабирование, нет похода в БД
на каждом запросе), и минус — отозвать выпущенный токен до его `exp` невозможно средствами
самого JWT.

Хранение JTI в Redis превращает access в **гибридный** токен: подпись остаётся stateless,
но появляется дешёвая (O(1) `EXISTS` в памяти) дополнительная проверка «не отозван ли он».
В момент logout удаляем ключ — следующий запрос с тем же токеном получит 401 от middleware,
хотя подпись и `exp` всё ещё валидны. Получаем «лучшее из двух миров»: stateless-проверку
подписи (не страдает производительность) и точечный отзыв (как у сессий).

**7. Какие данные из ЛР №3 (токены, сессии) безопасно кешировать, а какие нет?**

- ✅ **Безопасно:** JTI access-токенов (идентификатор, не материал подписи);
  идентификаторы активных сессий; флаги «токен отозван»; профиль пользователя без
  чувствительных полей.
- ⚠️ **С оговорками:** хеши refresh-токенов — кешировать можно, но смысла мало, если БД
  и так быстрая. Дополнительная поверхность атаки.
- ❌ **Нельзя:** сами refresh-токены / access-токены в открытом виде; пароли (даже хеши —
  никогда: ничто в кеше не оправдывает попадание материала пароля туда); полные claims с
  PII (телефон, email — только если действительно нужно и под строгим TTL); приватные ключи
  подписи JWT; OAuth refresh-токены провайдеров.

Правило простое: если содержимое ключа достаточно, чтобы получить доступ — это уже не кеш,
а secret store, и Redis для этого не подходит без отдельных мер (encryption-at-rest,
сетевая изоляция, аудит).

**8. Как повлияет перезапуск контейнера Redis на работу приложения (с учётом AOF/RDB или
без неё)?**

В `docker-compose.yml` Redis запущен с `--appendonly yes` (AOF). Это значит, что каждая
команда мутации пишется в `appendonly.aof` (с `fsync everysec` по умолчанию), и при
перезапуске Redis восстанавливает все данные. Volume `wp_labs_redis:/data` сохраняется
между ребутами контейнера, поэтому:

- **С AOF + persistent volume (наш случай):** перезапуск Redis потеряет максимум последнюю
  секунду записей. После старта все JTI и кеш будут на месте — пользователей не выкинет.
- **Без AOF / RDB (volatile):** все данные исчезают. Последствия:
  - кеш списков/профиля — холодный, первые запросы дольше, но это не сломает функционал
    (это и есть точка graceful degradation);
  - реестр JTI пуст → **если middleware fail-closed**, всех залогиненных пользователей
    выкинет на 401 до повторного логина. В нашей реализации **fail-open**: при отсутствии
    Redis-проверки токен считается валидным по подписи, пока ключ не появится снова —
    компромисс между безопасностью и UX. Сразу после старта Redis окажется без JTI, и
    мгновенный отзыв временно работать не будет — это известный edge case.

Поэтому AOF включать стоит, и именно для регистра JTI это важнее, чем для самого кеша.

**9. Что такое сериализация и зачем она нужна при сохранении объектов в Redis?**

Redis хранит байты — он не знает о Go-структурах. Чтобы положить туда `models.Item` или
`dto.PaginationResponse`, нужно превратить структуру в самодостаточную последовательность
байтов (сериализация) — обычно JSON, MessagePack, Protobuf или gob. При чтении выполняется
обратная операция (десериализация). В этом проекте выбран **JSON** через `encoding/json`:

- человекочитаемо в `redis-cli GET`, удобно при отладке лабы;
- автоматически уважает теги `json:"-"` у моделей, что **попутно решает безопасность**:
  поля `PasswordHash`/`Salt`/`YandexID`/`VkID` не попадают в Redis даже при сериализации
  всей модели `User` (см. кеш профиля в `AuthService.GetUserByID`);
- Protobuf был бы компактнее и быстрее, но требует схем и потерял бы читаемость.

**10. Как префиксы ключей помогают в управлении кешем?**

Префиксы дают логическое пространство имён в плоском keyspace Redis:

- **Изоляция приложений** на общем Redis-кластере: `wp:*`, `analytics:*`, `sessions:*`
  не пересекаются.
- **Массовые операции по подмножеству**: `SCAN MATCH wp:items:user:1:*` корректно достанет
  только items юзера 1, а не случайно совпавший ключ из другого модуля.
- **Безопасность массовых удалений**: `DelByPattern` в нашем проекте обязан получать
  префикс `wp:`, иначе бросает ошибку — это страховка от случайного `FLUSHALL`-подобного
  поведения по ошибочному паттерну.
- **Иерархия в имени**: `wp:auth:user:{id}:access:{jti}` сразу читается «модуль auth,
  пользователь N, его access-токен с JTI X» — это упрощает дебаг и операционную работу.
- **Метрики и алерты**: можно мониторить hit/miss rate по префиксам отдельно (`wp:items:*`
  кешируется хорошо, `wp:users:profile:*` плохо — повод подумать).

**11. Зачем защищать Redis паролем даже внутри закрытой Docker-сети?**

Defence-in-depth:

- **Любой другой контейнер в этой Docker-сети** может подключиться к Redis. Если в стэк
  попадёт скомпрометированный сервис (например, через уязвимость в зависимостях), без
  пароля он получит полный read/write доступ к Redis за один TCP-коннект, включая JTI
  всех пользователей.
- **Порт 6379 проброшен на хост** (`ports: 6379:6379` — для отладки лабы). Без пароля
  любой процесс на хосте, любая программа на машине разработчика подключается к Redis.
  На проде эту строку убирают, но забыть легко.
- **Docker-сеть — это L3-граница, а не L7-аутентификация**. Любой ошибочно открытый
  ingress, неправильно настроенный reverse-proxy или просто `docker run --network=...`
  с другим контейнером может «дотянуться» до Redis.
- **Команды разрушительной мощности** (`FLUSHALL`, `CONFIG SET dir`, `MODULE LOAD` при
  включённых модулях) при отсутствии AUTH доступны кому угодно.
- **Принцип нулевого доверия**: внутренний периметр не считается «безопасным» по умолчанию;
  каждое звено должно требовать аутентификацию.

Поэтому даже в учебной лабе `--requirepass` обязателен.

## Запуск без Docker (на хосте)

```bash
# 1. Поднять Postgres и Redis (Redis обязательно с паролем):
#    redis-server --requirepass yourpassword --appendonly yes
# 2. В .env прописать DB_*, REDIS_* (REDIS_HOST=localhost)
# 3. Сгенерировать docs
go install github.com/swaggo/swag/cmd/swag@v1.16.3
swag init -g main.go --parseDependency --parseInternal -o ./docs
# 4. Запустить
go run .
```

При отсутствии Redis приложение всё равно стартует — в логе будет
`[cache] Redis недоступен при старте: ... Кеш будет отключён`,
все методы кеша станут no-op, бизнес-логика будет ходить напрямую в БД.
